## 2. Java HotSpot VM体系结构

### 概观
Java HotSpot虚拟机是Sun用于Java平台的虚拟机。 它使用许多先进技术为Java应用程序提供最佳性能，并结合了最先进的内存模型，垃圾收集器和自适应优化器。 它是用高级的，面向对象的风格编写的，其特点是：
-  统一的对象模型
-  解释，编译和本地帧都使用相同的堆栈
-  基于本地线程的抢先式多线程
-  准确的分代和压缩垃圾收集
-  超快速的线程同步
-  动态去最佳化和积极的编译器优化
-  在VM启动时生成的特定于系统的运行时例程
-  编译器接口支持并行编译
-  运行时分析仅将焦点集中在“热门”方法上

JDK包含两种风格的虚拟机 - 客户端产品和针对服务器应用程序调整的虚拟机。 这两个解决方案共享Java HotSpot运行时环境代码库，但使用适合客户端和服务器的独特性能特征的不同编译器。 这些差异包括编译内联策略和堆默认值。

JDK在发行版中包含这两个系统，因此开发人员可以通过指定-client或-server来选择他们想要的系统。

尽管服务器和客户端虚拟机类似，但服务器虚拟机经过特别调整以最大化运行速度。它旨在用于执行长时间运行的服务器应用程序，这些应用程序需要最快的运行速度，而不仅仅是快速启动时间或更小的运行时内存占用量。

客户端VM编译器可用作旧版JDK使用的Classic VM和实时（JIT）编译器的升级。客户端VM为应用程序和小应用程序提供改进的运行时性能。 Java HotSpot客户端虚拟机经过特别调整，以减少应用程序启动时间和内存占用量，使其特别适合于客户端环境。一般来说，客户端系统对GUI更好。

客户端VM编译器不会尝试执行由服务器虚拟机中的编译器执行的许多更复杂的优化，但作为交换，它需要更少的时间来分析和编译一段代码。这意味着客户端VM可以更快地启动并且需要更小的内存占用量。

Server VM包含一个高级自适应编译器，支持通过优化C ++编译器执行的许多相同类型的优化，以及一些传统编译器无法完成的优化，例如跨虚拟方法调用的积极内联。这比静态编译器更具竞争力和性能优势。自适应优化技术在其方法中非常灵活，并且通常甚至胜过先进的静态分析和编译技术。

两种解决方案均可提供极其可靠，安全和可维护的环境，以满足当今企业客户的需求。

### 内存模型
##### 无柄物体(Handleless Objects)
在以前版本的Java虚拟机（如Classic VM）中，间接句柄用于表示对象引用。 虽然这使得在垃圾收集过程中更容易重新定位对象，但它代表了严重的性能瓶颈，因为访问Java编程语言对象的实例变量需要两个间接级别。

在Java HotSpot VM中，Java代码不使用句柄。 对象引用被实现为直接指针。 这提供了对实例变量的C速访问。 在内存回收期间重新定位对象时，垃圾回收器负责查找和更新对该对象的所有引用。

##### 双字对象头(双字对象头)
Java HotSpot VM使用两个机器字对象标题，而不是Classic VM中的三个字。 由于平均Java对象的大小很小，这对空间消耗具有显着的影响 - 对于典型应用程序，堆大小可节省约8％。 第一个标题字包含诸如身份哈希码和GC状态信息之类的信息。 第二个是对象类的引用。 对于数组大小，只有数组具有第三个头字段。

##### 反射数据被表示为对象(Reflective Data are Represented as Objects)

类，方法和其他内部反射数据直接表示为堆中的对象（尽管这些对象可能无法直接被Java技术程序访问）。 这不仅简化了VM内部对象模型，而且还允许通过用于其他Java编程语言对象的相同垃圾回收器来收集类。

##### 本地线程支持，包括抢占和多处理(Native Thread Support, Including Preemption and Multiprocessing)
每线程方法激活堆栈使用主机操作系统的堆栈和线程模型表示。 Java编程语言方法和本地方法共享相同的堆栈，允许C和Java编程语言之间的快速调用。 使用主机操作系统的线程调度机制支持完全抢先式Java编程语言线程。

使用本地操作系统线程和调度的一个主要优点是能够透明地利用本机操作系统多处理支持。 由于Java HotSpot虚拟机被设计为在执行Java编程语言代码时对先占和/或多处理引起的竞争条件不敏感，因此Java编程语言线程将自动利用本机操作系统提供的任何调度和处理器分配策略。


### 垃圾收集
Java HotSpot虚拟机内存系统的世代性提供了使用特定垃圾收集算法的灵活性，该算法适合于各种应用程序的需求。 Java HotSpot VM支持几种不同的垃圾收集算法，旨在满足不同的暂停时间和吞吐量要求。

##### 背景
程序员的Java编程语言的一个主要吸引力是它是第一个提供内置自动内存管理或垃圾收集（GC）的主流编程语言。 在传统语言中，动态内存使用显式分配/空闲模式进行分配。 实际上，这不仅是用传统语言编写的程序中的内存泄漏，程序错误和崩溃的主要来源，而且还是性能瓶颈和模块化可重用代码的主要障碍。 （如果没有模块之间的明确和难以理解的合作，确定跨越模块边界的空闲点几乎是不可能的。）在Java编程语言中，垃圾收集也是支持安全模型所需的“安全”执行语义的重要组成部分。


垃圾回收器会自动处理在场景后面释放未使用的对象内存，只有当它可以证明对象不再可供正在运行的程序访问时，它才会回收对象。 这个过程的自动化不仅彻底消除了因释放太少导致的内存泄漏，而且还消除了由于释放太多导致的程序崩溃和难以找到的参考错误。

传统上，与无明确模型相比，垃圾收集一直被认为是阻碍性能的低效过程。 实际上，采用现代垃圾收集技术后，性能已经有了很大的提高，整体性能实际上明显好于显式释放物体所提供的性能。

##### Java HotSpot垃圾收集器
除了包含下面介绍的最新功能之外，内存系统还被设计为一个干净的面向对象的框架，可以很容易地进行检测，实验或扩展以使用新的垃圾收集算法。

下面介绍Java HotSpot垃圾收集器的主要功能。 总体而言，这些功能非常适合需要尽可能高性能的应用程序，以及长期运行的应用程序，这些应用程序由于碎片而导致内存泄漏和内存不可访问性非常不理想。

##### 准确性

Java HotSpot垃圾收集器是一个完全准确的收集器。相比之下，许多其他垃圾收集器是保守的或部分准确的。虽然保守的垃圾收集可能很有​​吸引力，因为它很容易添加到没有垃圾收集支持的系统中，但它有一定的缺点。一般来说，保守的垃圾收集器容易发生内存泄漏，不允许对象迁移，并可能导致堆碎片。

保守的收集器并不知道所有对象引用的位置。因此，假设引用一个对象的内存字实际上是对象引用，它必须是保守的。这意味着它可能会产生某些类型的错误，例如将对象指针的整数混淆。看起来像指针的存储单元被视为指针 - 并且GC变得不准确。这有几个负面影响。首先，当出现这样的错误时（实际上并不常见），内存泄漏可能以应用程序员复制或调试实际上不可能发生的不可预测的方式发生。其次，保守收集器必须使用句柄间接引用对象 - 降低性能 - 或避免重新定位对象，因为重新定位无处理对象需要更新对对象的所有引用。如果收集器不确定明确参考是否是真实参考，则无法完成此操作。无法重新定位对象会导致对象内存碎片，并且更重要的是，会阻止使用下面介绍的高级世代复制收集算法。

由于Java HotSpot收集器是完全准确的，因此它可以提供几个强大的设计保证，以保守收集器无法做到：

所有不可访问的对象内存可以可靠地回收。

所有对象都可以重新定位，允许对象内存压缩，从而消除对象内存碎片并增加内存局部性。

准确的垃圾收集机制避免了意外的内存泄漏，启用了对象迁移，并提供了全面的堆压缩。 Java Hotspot VM中的GC机制可以很好地扩展到非常大的堆。


##### 世代复制集合(Generational Copying Collection)

Java HotSpot虚拟机采用了最先进的分代复制收集器，该收集器提供了两个主要优点：
-  与非代收藏家相比，大多数项目的分配速度和整体垃圾收集效率都有所提高
-  用户可感知垃圾收集暂停的频率和持续时间相应减少

世代收集器利用了这样的事实：在大多数程序中，绝大多数对象（通常超过95％）都非常短暂（例如，它们被用作临时数据结构）。 通过将新创建的对象分隔到对象托儿所中，分代收集器可以完成几件事情。 首先，因为新对象在对象托儿所中以堆栈式方式连续分配，所以分配变得非常快，因为它只涉及更新单个指针并执行对托儿所溢出的单个检查。 其次，在幼儿园溢出时，苗圃里的大部分物体已经死亡，垃圾收集者只需将其中的少数残存物体移动到其他地方，并避免在幼儿园为死亡物体进行填埋工作。


##### 并行年轻一代收集器(Parallel Young Generation Collector)

上面描述的单线程复制收集器虽然适用于许多部署，但可能成为扩展应用程序的瓶颈，否则这些应用程序将并行化以利用多个处理器。 为了充分利用多处理器机器上所有可用的CPU，Java HotSpot VM为年轻一代提供了可选的多线程收集器，在该收集器中，活动对象的跟踪和复制由多个并行工作的线程完成。 该实现经过仔细调整，以平衡所有可用处理器之间的收集工作，从而允许收集器扩展到大量处理器。 这减少了收集年轻空间的暂停时间并最大限度地提高垃圾收集吞吐量。 并行收集器已经通过包含超过100个CPU和0.5TB堆的系统进行测试。 并行年轻一代收集器是Server VM使用的默认垃圾收集算法。

在移动对象时，并行收集器会尝试将相关对象保持在一起，从而提高内存局部性和缓存利用率，并提高增强器的性能。 这是通过一阶复制对象来完成的。

并行收集器也更优化地使用可用内存。 它不需要保留旧对象空间的一部分，以保证复制所有活动对象的空间。 相反，它使用了一种新颖的技术来推测性地尝试复制对象。 如果旧的对象空间稀少，这种技术可以让收集器顺利切换以压缩堆，而不需要保存任何空间。 这可以更好地利用可用的堆空间。

最后，并行收集器能够响应应用程序的堆分配行为动态调整其可调参数，从而提高广泛应用程序和环境中的垃圾收集性能。 这意味着为客户减少手动调整工作。 此功能最初是在并行收集器中引入的，现在可用于许多其他垃圾收集算法。

与默认的单线程收集器相比，并行收集器的盈亏平衡点似乎在2到4个CPU之间，取决于平台和应用程序。 预计这将在未来的版本中进一步改进。

##### Mark-Compact旧对象收集器(Mark-Compact Old Object Collector)
尽管世代复制收集器能够高效地收集大多数死点对象，但较长寿命的对象仍会聚集在旧对象内存区域中。 有时，基于低内存条件或程序化请求，必须执行旧对象垃圾回收。 默认情况下，Java HotSpot VM使用标准的mark-compact收集算法，该算法从根中遍历整个活体对象图，然后扫过内存，压缩死对象留下的空隙。 通过压缩堆中的间隙，而不是将它们收集到空闲列表中，可以消除内存碎片，并且通过消除空闲列表搜索来简化旧对象分配。

##### 大多并行标记扫描收集器 Mostly Concurrent Mark-Sweep Collector
对于需要大堆的应用程序，由默认的旧代mark-compact收集器引发的收集暂停可能经常导致中断，因为应用程序线程被暂停了一段与堆大小成比例的时间段。 Java HotSpot VM为旧对象空间实现了一个可选的并发收集器，该收集器可以利用备用处理器周期（或备用处理器）来收集大堆，同时暂停应用程序线程很短的时间。这是通过在应用程序线程执行时执行大量的跟踪和清理工作来完成的。在某些情况下，由于某些处理器周期专用于并发采集活动，因此峰值应用吞吐量可能会有小幅下降;然而，平均和最坏情况下的垃圾收集暂停时间通常会减少一个或两个数量级，从而允许更平滑的应用程序响应，而不会在默认同步标记 - 紧凑算法在大堆上运行时出现突发。

#### 并行老一代收集器(Parallel Old Generation Collector)
当前版本的Java HotSpot VM为老一代引入了一个并行mark-compact收集器，旨在提高具有大堆的应用程序的可伸缩性。 在并发标记清除收集器着眼于减少暂停时间的情况下，并行旧收集器通过在停止世界暂停期间同时使用多个线程来收集旧一代来关注增加吞吐量。 并行旧收集器在内部使用许多新颖的技术和数据结构来实现高可扩展性，同时保留了精确的垃圾收集的好处，并且在收集周期期间最小化了簿记开销。


### 超快速线程同步

Java编程语言允许使用多个并发的程序执行路径 - 线程。 Java编程语言提供了语言级别的线程同步，这使得使用细粒度锁定来表示多线程程序变得很容易。 相对于Java编程语言中的其他微操作，以前的同步实现（如Classic VM中的实现）效率非常低，因此使用细粒度同步成为主要的性能瓶颈。

Java HotSpot虚拟机采用领先技术实现无争用和竞争同步操作，从而大幅提升同步性能。 动态构成大部分同步的非共同同步操作是通过超快速，恒定时间技术实现的。 通过最新的优化，在最好的情况下，即使在多处理器计算机上，这些操作本质上也是免费的。 竞争同步操作使用高级自适应旋转技术来提高吞吐量，即使对于存在大量锁争用的应用程序。 因此，同步性能变得如此之快，以至于绝大多数真实世界的程序都不是重要的性能问题。


### 64位架构

Java HotSpot VM的早期版本仅限于寻址四千兆字节的内存，即使在64位操作系统（如Solaris OE）上也是如此。 现在的服务器可以包含更多的内存，而四千兆字节对于桌面系统来说是很多的。 例如，Sun Fire E25K服务器每个域最多支持1.15 TB内存。 借助64位JVM，基于Java技术的应用程序现在可以利用此类系统的全部内存。

有几类应用程序可以使用64位寻址。 例如，那些在存储器中存储非常大的数据集。 应用程序现在可以避免从磁盘分页数据或从RDBMS中提取数据的开销。 这可能会导致此类应用程序的性能显着提升。

Java HotSpot VM现在是64位安全的，Server VM包含对32位和64位操作的支持。 用户可以分别使用命令行标志-d32或-d64选择32位或64位操作。 Java本地接口的用户需要重新编译他们的代码才能在64位虚拟机上运行它。

## 3.Java HotSpot编译器
大多数加速Java编程语言性能的尝试都着眼于应用为传统语言开发的编译技术。 
即时（JIT）编译器实质上是快速的传统编译器，可将Java技术字节码转换为本机机器码。 
运行在最终用户机器上的JIT实际上执行字节代码，并在第一次执行时编译每个方法。

但是，JIT编译有几个问题。 
首先，因为编译器在用户时间内运行在执行机器上，所以它在编译速度方面受到严重限制：如果编译速度不是很快，那么用户会感觉到程序启动或程序的一部分的显着延迟。 
这需要进行折衷，这使得执行高级优化变得困难得多，这通常会显着降低编译性能。

其次，即使JIT有时间进行完全优化，对于Java编程语言来说，这种优化对于C和C ++等传统语言来说效率较低。 有许多的原因：

Java语言是动态安全的，这意味着它确保程序不会违反语言语义或直接访问非结构化内存。这意味着必须经常执行动态类型测试（在投射时以及在存储到对象数组中时）。

Java语言分配堆中的所有对象，而C ++则是堆栈分配许多对象的对比。这意味着Java语言的对象分配率远高于C ++。另外，因为Java语言是垃圾收集的，所以它具有非常不同类型的内存分配开销（包括潜在的清除和写入屏障开销），而不是C ++。

在Java语言中，大多数方法调用都是虚拟的（可能是多态的），并且比C ++更频繁地使用。这意味着不仅方法调用性能更占优势，而且静态编译器优化（尤其是全局优化，如内联）对于方法调用来说更难执行。许多传统优化在调用之间最为有效，而且Java语言中调用之间的距离缩短可能会显着降低此类优化的有效性，因为它们只有较小的代码段可供使用。

由于强大的动态加载类的能力，基于Java技术的程序可以随时更改。这使得执行多种类型的全局优化变得更加困难。编译器不仅必须能够检测何时这些优化由于动态加载而变得无效，而且还能够在程序执行期间撤销或重做这些优化，即使它们涉及堆栈上的活动方法。这必须以任何方式完成而不会影响或影响基于Java技术的程序执行语义。

因此，任何试图在Java语言性能方面取得根本进展的尝试都必须为这些性能问题提供非传统的答案，而不是盲目应用传统的编译器技术。

Java HotSpot VM体系结构通过使用自适应优化技术来解决上述Java语言性能问题。

### 热点检测
自适应优化通过利用有趣的程序属性来解决JIT编译的问题。几乎所有的程序都花费绝大部分时间来执行少数代码。 Java HotSpot虚拟机不是按照方法编译方法，而是立即使用解释器运行程序，并在运行时分析代码以检测程序中的关键热点。然后它将全局本机代码优化器的注意力集中在热点上。通过避免编译不常执行的代码（大部分程序），Java HotSpot编译器可以更多地关注程序的性能关键部分，而不必增加整个编译时间。随着程序运行，此热点监控将继续动态进行，以便它可以根据用户的需求随时调整其性能。

这种方法的一个微妙而重要的好处是，通过延迟编译直到代码已经执行了一段时间（以机器时间而不是用户时间来测量），可以在代码被使用的方式上收集信息，然后被利用进行更智能的优化。内存占用减少。除了收集程序中热点的信息之外，还会收集其他类型的信息，例如关于虚拟方法调用的调用者 - 被调用者关系的数据。

### 方法内联
Java编程语言中虚拟方法调用的频率是一个重要的优化瓶颈。一旦Java HotSpot自适应优化器在执行程序热点期间收集了信息，它不仅将热点编译为本地代码，而且还对该代码执行广泛的方法内联。

内联有重要的好处。它大大降低了方法调用的动态频率，从而节省了执行这些方法调用所需的时间。但更重要的是，内联为优化器生成更大的代码块。这创造了一种显着提高传统编译器优化效率的情况，克服了提高Java编程语言性能的主要障碍。

内联与其他代码优化是协同的，因为它使它们更有效。随着Java HotSpot编译器的成熟，在大型内联代码块上运行的能力将为将来进行更多高级优化打开大门。


### 动态优化
虽然上一节介绍的内联是一种重要的优化，但对于像Java语言这样的动态面向对象语言来说，传统上来说执行起来非常困难。此外，虽然检测热点并将它们调用的方法内联足够困难，但仍不足以提供完整的Java编程语言语义。这是因为用Java语言编写的程序不仅可以即时更改方法调用的模式，还可以将新的Java代码动态加载到正在运行的程序中。

内联是基于全局分析的一种形式。动态加载显着增加了内联的复杂性，因为它改变了程序中的全局关系。一个新类可能包含需要在适当位置内联的新方法。因此，即使执行热点代码，Java HotSpot VM也必须能够动态去优化（如有必要，然后重新优化）以前优化的热点。没有这种能力，通用内联不能在基于Java技术的程序中安全地执行。

Java HotSpot Client和Server编译器都完全支持动态去最佳化。这使得积极和乐观的优化以及其他技术（如后面将介绍的全速调试）成为可能。

### Java HotSpot客户端编译器
Java HotSpot客户端编译器是一个简单，快速的三阶段编译器。 在第一阶段，独立于平台的前端从字节码构建高级中间表示（HIR）。 HIR使用静态单一赋值（SSA）形式来表示值，以便更容易地实现某些优化，这些优化是在IR构建期间和之后执行的。 在第二阶段，特定于平台的后端从HIR生成低级中间表示（LIR）。 最后阶段使用线性扫描算法的定制版本在LIR上执行寄存器分配，在LIR上执行窥视孔优化并从其生成机器码。

重点在于从字节码中提取和保留尽可能多的信息。 客户端编译器侧重于本地代码质量，并且只做很少的全局优化，因为这些编译时间通常是最昂贵的。

### Java HotSpot服务器编译器
服务器编译器针对典型服务器应用程序的性能配置文件进行了调整。 

Java HotSpot Server Compiler是一个高端的全面优化编译器。

它使用先进的基于静态单指派（SSA）的IR进行优化。

优化器执行所有经典优化，包括死代码消除，循环不变提升，常见子表达式消除，常量传播，全局值编号和全局代码运动。

它还具有Java技术特有的优化功能，如空值检查和范围检查消除以及异常抛出路径的优化。

寄存器分配器是全局图着色分配器，充分利用RISC微处理器中常见的大型寄存器集。编译器具有高度的可移植性，依靠机器描述文件来描述目标硬件的各个方面。

虽然JIT标准的编译器速度很慢，但它仍然比传统的优化编译器快得多，并且改进的代码质量通过减少编译代码的执行时间来缩短编译时间。

### 编译器优化

Java HotSpot编译器支持一系列高级优化，以实现传统直线程序以及面向对象编程风格的高性能。 其中一些优化包括：

深入内嵌和内嵌潜在的虚拟调用：如上所述，客户端和服务器编译器使用方法内联以及全局分析和动态去优化，以实现深入内联，从而消除大量的方法调用开销。

快速instanceof / checkcast：Java HotSpot VM和编译器支持一种新颖的技术，用于加速Java编程语言为了类型安全而频繁使用的动态类型测试。这进一步降低了面向对象风格编程的运行时成本。

范围检查消除：Java编程语言规范要求在每个数组访问时执行数组边界检查。编译器可以证明用于数组访问的索引处于边界内时，可以消除索引边界检查。

循环展开：服务器虚拟机具有循环展开功能，这是一种标准的编译器优化，可实现更快的循环执行。循环展开增加了循环体的大小，同时减少了迭代次数。循环展开也增加了其他优化的有效性。

反馈定向优化：在编译Java字节码以优化机器代码之前，服务器VM在解释器中执行程序的大量分析。这个分析数据为编译器提供了更多关于正在使用的数据类型，通过代码的热路径以及其他属性的信息。编译器使用这些信息在某些情况下更积极和乐观地优化代码。如果代码的某个假定属性在运行时被违反，则代码将被去优化，然后重新编译并重新优化。

## 4.Java HotSpot VM的高级功能
Java HotSpot VM支持许多高级功能，以实现高可扩展性，高性能和企业级可靠性，可用性和可维护性。

#### 可扩展性
Java HotSpot VM最近添加了称为人体工程学的自动尺寸调整和自适应机制。目前，人体工程学体现在两个主要领域。首先，根据机器的物理配置（例如考虑处理器数量和可用物理内存），客户端VM或服务器VM将自动选择。特别是，对于处理器数量较多且RAM数量较多的计算机，将选择服务器虚拟机，并且会自动为此类硬件上合理的服务器端应用程序选择垃圾收集堆的大小。其次，Java HotSpot虚拟机中的垃圾收集算法现在是自我调整的，因此不再需要明确指定年轻人和老一代的相对大小。垃圾收集器将自行调整以提高应用程序的吞吐量并减少暂停时间。人体工程学技术可自动提高服务器端应用程序的可扩展性，并计划在未来版本中开展更多工作。

#### 性能
除了Java HotSpot VM架构支持的核心面向对象优化外，VM和Sun Java运行时环境还支持一些其他关键性能优化：

快速反射：Java库现在为常用的反射对象（如Methods和Constructors）生成字节码存根。这种技术将反射调用公开给Java HotSpot编译器，产生更高的性能，并且在某些情况下使用服务器虚拟机，可以完全消除与反射调用相关的开销。这在反射密集型代码中提供了显着的加速，例如在序列化，RMI和CORBA代码环境中使用的代码。
新的I / O优化：Java HotSpot编译器专门处理New I / O Buffer对象上的操作，为get和put方法调用生成高质量的机器代码。结合新I / O API提供的网络和文件I / O的大量性能和可伸缩性改进，Java编程语言应用程序现在可以实现与使用C和C ++编码的应用程序相似的吞吐量。新的I / O缓冲区优化也适用于其他问题领域，如3D图形，在Java平台和外部世界之间传输大量数据。

#### 可靠性，可用性和可服务性
目前版本的Java HotSpot VM是迄今为止最可靠的。 最近发布的VM基于Sun Microsystems，Inc.执行各种大型应用程序，为企业应用程序的可靠性和可用性设定了新的记录。

Java HotSpot VM包含Java虚拟机工具接口（JVM TI）的参考实现。 该接口允许分析器，调试器和监视器等工具观察和控制JVM。 包括的功能是：

全速调试：Java HotSpot VM利用动态去最佳化技术来全速支持应用程序的调试。在较早的Java虚拟机实现中，当启用调试支持时，应用程序仅在解释器中运行。在调试场景中启用Java HotSpot编译器可以极大地提高性能，并且在很多情况下，可以在始终启用调试支持的情况下运行，以提高应用程序的可用性。另外，抛出异常时可以触发调试器的启动。
HotSwap支持：Java HotSpot VM的面向对象体系结构支持高级功能，例如即时类重新定义或“HotSwap”。此功能提供了通过调试器API在正在运行的应用程序中替换已修改的代码的功能。 HotSwap为Java平台调试器体系结构增加了功能，使得在调试器控制下在执行期间可以更新类。它还允许通过热插入已插入分析代码的方法版本来执行分析操作。


Java HotSpot VM中的几项附加功能改进了Java编程语言应用程序的开发和可维护性。

JNI错误检查：可用于执行额外的JNI检查的命令行选项-Xcheck：jni。这使得可以在开发期间运行对参数有效性的检查，在部署之前可以检测到它们并减缓生产运行。具体来说，Java HotSpot VM在处理JNI请求之前验证传递给JNI函数的参数以及运行时环境数据。遇到的任何无效数据表示本机代码中存在问题，并且在这种情况下，VM将以致命错误终止。

错误报告：如果JVM检测到本机代码崩溃（例如开发人员编写的JNI代码），或者JVM本身崩溃，则它将打印并记录关于崩溃的调试信息。此错误消息通常会包含信息，例如函数名称，库名称，源文件名称和发生错误的行号。结果是开发人员可以更轻松高效地调试他们的应用程序。如果错误消息表明JVM代码本身存在问题，则它允许开发人员提交更准确且有用的错误报告。

信号链工具：信号链使Java平台能够更好地与安装自己的信号处理程序的本机代码进行互操作。该设施可在Solaris OE和Linux平台上运行。引入信号链功能是为了弥补以前版本的Java HotSpot VM中的信号处理问题。在版本1.4之前，Java HotSpot VM将不允许应用程序安装的信号处理程序用于某些信号，例如SIGBUS，SIGSEGV或SIGILL，因为这些信号处理程序可能会与Java HotSpot VM内部使用的信号处理程序发生冲突。


## 2.对软件可重用性的影响
面向对象编程的主要优点是可以通过为软件重用提供强大的语言机制来提高开发效率。 然而，在实践中，这种可重用性很难达到。 这些机制的广泛使用可以显着降低性能，这使得程序员有节制地使用它们。 Java HotSpot技术令人惊讶的副作用是它显着降低了性能成本。 Sun相信，这将对如何开发面向对象软件产生重大影响，允许公司第一次充分利用面向对象的可重用机制，而不会影响其软件的性能。

这种效果的例子很容易得出。 对使用Java编程语言的程序员的调查很快就会发现，许多人避免使用完全虚拟的方法（并且编写更大的方法），因为他们认为每个虚拟方法调用都会导致严重的性能损失。 虚拟方法（如Java编程语言中不是静态或最终的方法）的无处不在的细粒度使用对于构建高度可重用的类非常重要，因为每个此类方法的作用都像一个允许新子类修改的钩子 超类的行为。

由于Java HotSpot虚拟机可以自动内联绝大多数虚拟方法调用，因此性能损失大大降低，并且在很多情况下完全被取消。

很难夸大这种影响的重要性。 它有可能从根本上改变面向对象代码的编写方式，因为它显着改变了使用重要可重用机制的性能权衡。 此外，很明显，随着面向对象编程的成熟，趋向于更细粒度的对象和更细粒度的方法。 这一趋势强烈指出未来编码风格中虚拟方法调用频率的增加。 随着这些更高级的编码风格变得流行，Java HotSpot技术的优势将变得更加明显。


## 6. 总结

Java HotSpot VM为Java应用程序提供最佳性能，提供高级优化，垃圾回收和线程同步功能。 另外，VM提供调试功能，旨在提高基于Java技术的应用程序的整体可靠性，可用性和可维护性。 Java HotSpot VM为客户端和服务器环境提供单独的编译器，以便应用程序可根据其目标部署环境进行优化。 通过提供64位Java HotSpot服务器编译器，可伸缩性得到显着增强。

借助Java HotSpot VM，客户端应用程序启动速度更快，所需内存占用空间更小，而服务器应用程序在长期运行中可实现更好的持续性能。 两种解决方案都能提供极其可靠，安全和可维护的环境，以满足当今企业客户的需求。
