# 垃圾收集器人体工程学(Garbage Collector Ergonomics)

以下更改将在J2SE 5.0中生效

### 1.运行的服务器VM服务器级的机器，垃圾收集器（GC）已经从先前的串行收集器改变（-XX：+ UseSerialGC）到并行收集器（-XX：+ UseParallelGC）。您可以通过使用-XX修改这个默认+ UseSerialGC命令行选项java命令。

### 2.在运行具有并行垃圾收集器（-XX：+ UseParallelGC）的VM（客户机或服务器）的服务器级机器上，初始堆大小和最大堆大小已更改如下。

-  初始堆大小：
机器物理内存的1/64更大或合理的最小值。 在J2SE 5.0之前，默认的初始堆大小是一个合理的最小值，因平台而异。 您可以使用-Xms命令行选项覆盖此默认值。
-  最大堆大小
较小的物理内存的1/4或1GB。 在J2SE 5.0之前，默认的最大堆大小为64MB。 您可以使用-Xmx命令行选项覆盖此默认值。

针对堆大小给出的边界和分数对于J2SE 5.0是正确的。 随着计算机功能越来越强大，它们在后续版本中可能会有所不同。

### 3. 并行垃圾收集器（UseParallelGC）会在收集少量堆的过量时间内抛出内存不足异常。 为避免此异常，您可以增加堆的大小。 您还可以设置参数-XX：GCTimeLimit =时间限制和-XX：GCHeapFreeLimit =空间限制其中：

时间限制：
垃圾收集花费的时间上限（以总时间百分比表示）（默认值为98）。

空间限制：
他在垃圾回收期间释放的空间量的下限为最大堆的百分比（默认值为2）。

### 4. 默认使用的-XX：+ UseAdaptiveSizePolicy与-XX：+ UseParallelGC垃圾收集器的实现已更改为考虑三个目标：
-  一个期望的最大GC暂停目标
-  所需的应用程序吞吐量目标
-  最小的足迹

实施检查（按此顺序）：
1.  如果GC暂停时间大于暂停时间目标，则缩小代码大小以更好地实现目标。
2.  如果暂停时间目标得到满足，则考虑应用程序的吞吐量目标。 如果应用程序的吞吐量目标未得到满足，则应增加各代的大小以更好地实现目标。
3.  如果暂停时间目标和吞吐量目标都得到满足，那么世代的大小会减少以减少占用空间。

参数：
```-XX:MaxGCPauseMillis=nnn```
向虚拟机提示暂停nnn毫秒或更少的时间是需要的。 VM将调整java堆大小和其他GC相关参数，以试图使GC引起的暂停短于nnn毫秒。 请注意，这可能会导致虚拟机降低整体吞吐量，并且在某些情况下，虚拟机将无法满足所需的暂停时间目标。

默认情况下没有暂停时间目标。 暂停时间目标能够达到的程度有一定的限制。 GC的暂停时间取决于堆中的实时数据量。 小数据库和主要数据集以不同的方式依赖于实时数据量。 该参数应谨慎使用。 值太小会导致系统花费过多时间进行垃圾收集。

``` -XX:GCTimeRatio=nnn ```
向虚拟机提示在收集器中应用不超过1 /（1 + nnn）的应用程序执行时间是可取的。
例如-XX：GCTimeRatio = 19设定GC总时间的5％和吞吐量目标为95％的目标。 也就是说，应用程序应该比采集器多19倍的时间。

默认情况下，该值为99，这意味着应用程序应至少获得收集器99倍的时间。 也就是说，收集器的运行时间不应超过总时间的1％。 这被选为服务器应用程序的不错选择。 值太高会导致堆的大小增大到最大值。

### 建议的战略
除非您知道堆大于默认最大堆大小，否则不要选择堆的最大值。 选择一个足以满足您的应用程序的吞吐量目标。

在理想情况下，堆将增长到一个可支持所选吞吐量目标的值（小于最大值）。

如果堆增长到最大值，则吞吐量不能在该最大值内得到满足。 将最大堆设置为尽可能大，但不大于平台上物理内存的大小，然后再次执行该应用程序。 如果仍然无法达到吞吐量目标，那么平台上的可用内存太高。

如果吞吐量目标可以达到，但是暂停时间太长，请选择一个暂停时间目标。 这可能意味着您的吞吐量目标将无法满足，因此请选择适合应用程序的折衷值。
