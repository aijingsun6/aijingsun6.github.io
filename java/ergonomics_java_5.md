# java 5.0 虚拟机中的人体工程学
[原文档地址](http://www.oracle.com/technetwork/java/ergo5-140223.html)

### 1.介绍
在J2SE平台版本5中，对于某些平台，垃圾收集器，堆大小和运行时编译器的默认选择已更改。 这些新的选择更符合不同类型应用的需求，同时需要较少的命令行调整。 另外，为吞吐量垃圾回收器添加了不同的调优堆的方式。 这种新的调整允许用户指定所需的行为，并依赖于垃圾回收器来动态调整堆的大小以满足指定的行为。 平台相关的默认选择和使用所需行为的垃圾收集调优的组合在本文档中称为“人体工程学”。

该文档是与5.0 Java™虚拟机的文件“调谐垃圾收集”的文件。 假设熟悉该调优文档，虽然熟悉1.4.2 Java虚拟机的该文档的先前版本应该足够了。
### 2.垃圾收集器，堆和运行时编译器
在J2SE平台版本5.0中，一类称为服务器级机器的机器已被定义为机器

-  2个以上物理处理器
-  2 GB或更多GB的物理内存

默认情况下，在服务器级机器上选择以下选项。

-  吞吐量垃圾回收器
-  堆大小
-  物理内存的最初堆大小为1/64，高达1Gbyte
-  物理内存的最大堆大小可达1Gbyte
-  服务器运行时编译

服务器级机器的定义适用于运行Windows™操作系统版本的32位平台的所有平台。 在所有其他平台上，默认值与版本1.4.2的默认值相同。 下面是一个表，显示了在差异平台上为运行时编译器所做的选择。

### 3.基于行为调整
在J2SE平台版本5.0之前，对垃圾收集的调整主要包括指定堆的总体大小和可能的代数。调整垃圾收集的其他控制措施包括年轻一代幸存者空间的大小以及从一代到老一代的晋升门槛。调整需要一系列实验，具有不同的这些参数值，使用专门的工具或者只是很好的判断来决定垃圾收集表现良好。

在版本5.0中，已经介绍了基于应用程序所需行为的两个参数。这些是

-  最大暂停时间目标

-  应用吞吐量目标。

设置这些目标可以用来调整垃圾收集。应该强调的是，目标不能总是得到满足。该应用程序需要足够大的堆以至少拥有所有的实时数据。这个最小堆大小我排除了达到这些目标。

#### 3.1最大暂停时间目标
暂停时间是垃圾收集器停止应用程序并恢复不再使用的空间的持续时间。最大暂停时间目标的目的是限制这些暂停中最长的时间。平均停电时间和该平均值的差异由垃圾收集器维护。平均值来自执行开始时的加权，因此最近的暂停更重。如果平均加上暂停时间的方差大于最大暂停时间目标，那么垃圾收集器认为目标没有得到满足。

使用命令行标志指定最大暂停时间目标

-XX：MaxGCPauseMillis = <NNN>

这被解释为对垃圾回收器的提示，希望暂停时间<nnn>毫秒或更少。垃圾回收器将调整Java堆大小和其他垃圾回收相关参数，以避免垃圾回收暂停时间短于<nnn>毫秒。默认情况下，没有最大暂停时间目标。这些调整可能会导致垃圾收集器发生频率更高，从而降低了应用程序的总体吞吐量。在某些情况下，无法满足所需的暂停时间目标。
 
吞吐量收集器是一代人收藏家，因此，对于年轻一代和老一代来说，这是一个单独的收藏。平均值和方差分别保留给每一代。最大暂停时间目标分别应用于平均加上每一代的集合的方差。每一代人可能会单独失败，以达到暂停时间目标。
 
#### 3.2吞吐量目标
吞吐量目标是根据收集垃圾的时间和垃圾回收以外的时间（简称应用时间）来衡量的。目标由命令行标志指定

-XX：GCTimeRatio = <NNN>

垃圾收集时间与申请时间的比例是
 
1 /（1 + <nnn>）
 
例如-XX：GCTimeRatio = 19设置了垃圾回收总时间的1/20或5％的目标。
 
垃圾收集的时间是年轻一代和老一代收藏的总和时间。如果吞吐量目标未得到满足，则增加代数的大小，以增加应用程序在集合之间运行的时间。
 
#### 3.3脚印目标
如果满足吞吐量和最大暂停时间目标，则垃圾收集器将减小堆的大小，直到达不到目标之一（总是吞吐量目标）为止。然后解决未达到的目标。

### 4.调整策略
除非您知道需要一个大于默认最大堆大小的堆，否则不要为堆选择最大值。选择对您的应用程序足够的吞吐量目标。
 
堆将增长或缩小到将支持所选吞吐量目标的大小。预期在初始化期间和在应用程序行为发生变化时，堆大小的一些振荡。
 
如果堆增长到最大值，在大多数情况下，这意味着在最大堆大小内无法满足吞吐量目标。将最大堆大小设置为接近平台上物理内存总量的值，但不会导致应用程序交换。再次执行应用程序如果吞吐量目标仍未达到，那么应用程序时间的目标对于平台上的可用内存来说太高了。
 
如果吞吐量目标可以满足，但暂停时间过长，请选择最大暂停时间目标。选择最大暂停时间目标可能意味着您的吞吐量目标将不会被满足，因此可以为应用程序选择可接受的折衷值。
 
垃圾收集器尝试满足竞争目标时，堆的大小是典型的。即使应用程序达到稳定状态也是如此。实现吞吐量目标的压力（可能需要较大的堆）与最大暂停时间和最小占用空间（可能需要小堆）的目标相竞争。

### 5.其他变化
线程本地分配缓冲区
线程本地分配缓冲区的大小已经被重新实现了。 当线程本地分配缓冲区填满时，新缓冲区的大小取决于线程的分配模式。 分配更多内存的线程将获得更大的缓冲区。
