## 显示垃圾回收 VS 自动垃圾回收{#c2}
内存管理是识别何时不再需要分配对象的过程，取消分配
（释放）这些对象使用的内存，并使其可用于后续分配。 在某些
编程语言，内存管理是程序员的责任。 这个任务的复杂性
导致许多常见的错误，可能导致意外或错误的程序行为和崩溃。 作为一个
结果是，大部分的开发人员经常花时间调试并尝试纠正这种错误。

通常在具有显式内存管理的程序中出现的一个问题是悬念引用(dangling references)。 
它是可能释放一些其他对象仍然具有引用的对象使用的空间。 如果对象
使用（悬挂）引用尝试访问原始对象，但是空间已被重新分配给新的对象
对象，结果是不可预测的，而不是什么意图。

显式内存管理的另一个常见问题是内存泄漏。 这些泄漏发生在内存时已分配，不再引用但未释放。 
例如，如果您打算释放a使用的空间链接列表，但是您错误地只是释放列表的第一个元素，剩下的列表元素不再被引用，但它们不符合程序的范围，既不能被使用也不能恢复。 
如果发生足够的泄漏，它们可以持续消耗内存，直到所有可用内存耗尽。

内存管理的另一种方法现在被普遍利用，特别是大多数现代的面向对象的语言，被称为垃圾回收器的程序自动管理。 自动内存管理可以增加接口的抽象和更可靠的代码。

垃圾收集避免了悬挂指针问题，因为一个仍然被引用的对象永远不会被垃圾收集，所以不会被认为是可回收的。 
垃圾收集也解决了空间泄漏因为它自动释放所有不再被引用的内存。

## 垃圾回收机制观念
垃圾回收器的责任包括：
-  分配内存
-  保证任何引用在内存中
-  恢复在执行代码中不能从引用中访问的对象使用的内存

被引用的对象被称为活的(alive)的。 不再引用的对象被认为是死亡(dead)的称为垃圾。 查找和释放（也称为回收）这些对象使用的空间的过程被称为垃圾收集。

垃圾收集解决了许多但不是全部的内存分配问题。 例如，您可以创建对象无限期地继续引用它们，直到没有更多的可用内存。 垃圾收集也是一个复杂的任务花费自己的时间和资源


管理内存，分配，回收内存的精算算法由垃圾回收器还有隐藏的程序员负责，内存通常由大量的内存池分配的，我们称之为堆

垃圾回收的时间取决于垃圾回收器，通常整个堆或者子部分填满的时候，或者达到一个百分比的阀值的时候会触发垃圾回收。

完成分配请求的任务，其中涉及到找到一定大小的未使用的内存块
堆，是一个困难的。 大多数动态内存分配算法的主要问题是避免
分散（见下文），同时保持分配和释放的效率

### 理想的垃圾收集器特性
垃圾收集机必须既安全又全面。 也就是说，实时数据不能被错误地释放，
并且垃圾不应该超过少量的收集周期而不被无人认领

当然我们还希望垃圾回收器能有效的操作，不能使应用程序出现卡顿。然而，与大多数与计算机相关的系统一样，往往有对时间，空间，频率的权衡。例如，如果堆很小，那么堆很快就满了，垃圾回收的频率将加快。
相反，如果堆很大，需要很长的时间来讲堆填满，但是垃圾回收的时间比较长。

另一种理想的垃圾回收器的特征是分段的限制。当内存为
垃圾对象被释放，自由空间可能会出现在各个区域的小块中，从而可能没有
在任何一个连续区域中有足够的空间用于分配大对象。 一种方法
消除碎片被称为压缩，在各种垃圾收集器设计选择之间进行讨论

可扩展性也很重要。 分配不应成为多线程的可扩展性瓶颈
多处理器系统上的应用程序和收集也不应该是这样的瓶颈
### 设计选择
在设计或选择垃圾收集算法时，必须做出很多选择：
-  串行与并行
  
  通过串行收集，一次只发生一件事情。 例如，即使是多个CPU
可用，只有一个用于执行收集。 当使用并行收集时，任务
垃圾收集被分成几部分，这些子部分被同时执行，不同
的CPU。 并行操作可以更快地完成收集，并以牺牲为代价
一些额外的复杂性和潜在的分裂。
-  并行与停止世界

当执行世界垃圾收集时，完全执行应用程序
在收集期间暂停。 或者，可以执行一个或多个垃圾收集任务
同时，也就是与应用程序同时进行。 通常，并发垃圾收集器
它的大部分工作同时进行，但也可能偶尔也要做一些短暂的停顿
暂停。 由于堆是，世界垃圾收集比并发收集更简单
在收集过程中冻结和物体没有变化。 它的缺点是它可能是
某些应用程序暂停不适用。 相应地，暂停时间较短
垃圾收集是同时进行的，但收集者必须特别小心，因为它正在运行
可能由应用程序同时更新的对象。 这增加了一些开销
影响性能并需要较大堆大小的并发收集器。
-  压缩与非压缩复制

垃圾回收器确定了内存中的哪些对象是生存的，哪些是垃圾？
可以压缩内存，将所有活动对象一起移动并完全回收
剩余记忆压实后，首先可以轻松快速地分配一个新对象
位置。可以使用简单的指针来跟踪可用于对象的下一个位置
分配。与压实收集器相反，非压实收集器释放空间
就地使用垃圾对象，即它不会移动所有活动对象以创建大型回收
区域与压实收集器相同。好处是更快完成垃圾
收集，但缺点是潜在的分裂。一般来说，分配更昂贵
从一个堆就地解除分配而不是一个压实的堆。可能需要搜索
堆的连续区域的内存足够大以容纳新对象。三分之一
替代方案是复制收集器，它将活动对象复制（或撤离）到不同的存储区域。
好处是，源区可以被认为是空的，可用于快速和容易
后续分配，但缺点是复制所需的额外时间和额外的时间
可能需要的空间

### 性能指标
利用几个指标来评估垃圾回收器性能，包括
-  吞吐量：未用于垃圾收集的时间与总时间的百分比
-  垃圾收集开销：吞吐量的倒数，即占用的总时间的比垃圾收集的时间
-  暂停时间：执行垃圾回收，系统暂停的时间
-  收集频率：垃圾回收的频率
-  足迹：大小的度量，比如堆大小
-  提示：当对象变成垃圾和内存变成可用内存的时间

交互式应用程序可能需要较低的暂停时间，而总体执行时间对于非交互程序更为重要。 实时应用程序将要求垃圾回收的上限很小
在任何时期停留和在收集者中花费的时间比例。 
小足迹可能是主要的关注在小型个人计算机或嵌入式系统中运行的应用程序
