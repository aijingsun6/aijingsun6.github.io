# 时间校正(time_correction)
## 新的时间拓展函数
从OTP18开始，关于时间的函数被拓展了，包含一些新的API接口，时间扭曲模式（Time Warp Mode）可以在系统时间改变时候改变系统的行为

虽然旧的API仍然是可以用的，***但是强烈建议您使用新的API***,来替代 ```erlang:now/0```,```erlang:now/0```被弃用了，这个函数存在瓶颈

通过使用新的时间函数API,你可以获得可拓展性还有性能更优的好处，你可以使用双时间包裹模式(multi-time warp mode)来提高时间度量的准确度还有精度

## 一些术语(Terminology)
为了方便沟通描述，我们这节会定义一些术语(Terminology),比如
erlang系统时间(erlang system time)，操作系统时间(os system time)，erlang 单调递增时间，系统单调递增时间，时间扭曲(time warp)，还有一些全球公认的术语

### 单调递增(Monotonically Increasing)
在单调递增的值序列中，具有前驱的所有值都大于或等于其前一个值。
### 严格单调递增(Strictly Monotonically Increasing)
在严格单调递增的值序列中，具有前驱的所有值都大于其前一个值。
### UT1
世界时。 UT1基于地球的旋转，概念上意味着经度为0°的太阳时。
### UTC
UTC几乎与UT1对齐。 但是，UTC使用秒的SI定义，其长度与UT1使用的秒的长度不完全相同。 这意味着UTC从UT1缓慢漂移。 为了使UTC与UT1保持同步，插入了闰秒，并且可能还会删除。 也就是说，UTC日可以是86400,86401或86399秒。
### POSIX Time
从1970-01-01 00:00:00 到现在的时间，按秒计算
### 时间分辨率（时间分辨率）
读取时间值时可以区分的最短时间间隔。
### 时间精度（Time Precision）
读取时间值时可以重复且可靠地区分的最短时间间隔。 精度受分辨率的限制，但分辨率和精度可能会有很大差异。
### 时间准确性（Time Accuracy）
时间值的正确性。
### 时间扭曲（Time Warp）
时间扭曲是一个向前或向后的飞跃。 也就是说，时间扭曲之前和之后所取的时间值的差异与实际的经过时间不对应。
### OS System Time
操作系统的POSIX时间，得到这个值可以通过```os:system_time()```来获得，这个可能是一个准确的时间戳，通常可以向后和向前调整该时间而没有限制。时间那就去可以被观测到
如果想得到更多的信息，请调用API```erlang:system_info(os_system_time_source)```
### OS 单调递增时间
操作系统提供的单调增加的时间。 这个时间没有跳跃并且具有相对稳定的频率，尽管不完全正确。 但是，如果系统暂停，则OS单调时间停止并不罕见。 此时间通常会增加，因为某些未指定的时间点与OS系统时间无关。 这种类型的时间不一定由所有OS提供。
如果想获得更多信息，请调用API```erlang:system_info(os_monotonic_time_source).```

### Erlang 系统时间
使用API```erlang:system_time().```来获得POSIX时间
此时可能是也可能不是POSIX时间的准确视图，可能与OS系统时间一致，也可能不一致。 运行时系统用于对齐两个系统时间。 根据所使用的时间扭曲模式，可以通过让Erlang系统时间执行时间扭曲来实现。

### Erlang 单调递增时间
Erlang运行时系统提供的单调增加的时间。 Erlang单调时间从一些未指定的时间点开始增加。调用API ```erlang:monotonic_time().```可以获得相关信息

erlang单调递增时间的准确度和精度取决于 Os单调递增时间(OS monotonic time)的准确度和精度，Os系统时间(OS system time)的准确度和精度，时间扭曲模式(time warp mode)

在没有OS单调时间的系统上，Erlang单调时间保证单调性，但不能给出其他保证。 对Erlang单调时间进行的频率调整取决于所使用的时间扭曲模式。

在运行系统内部，erlang单调递增时间才是时间引擎，处理有关时间的一切。所有的timer，除非内置的receive...after...的timer,都是使用单调递增时间来度量，即使是erlang系统时间，都是基于单调递增时间(Erlang monotonic time),通过添加当前时间偏移的当前Erlang单调时间，您将获得当前的Erlang系统时间

通过使用API```erlang:time_offset/0.```获取当前时间的偏移(offset)

## 介绍
时间对于Erlang程序至关重要，更重要的是，正确的时间对于Erlang程序至关重要。由于Erlang是一种具有软实时属性的语言，我们可以在程序中表达时间，因此虚拟机和语言必须注意所谓的正确时间和时间函数的行为方式。

当设计Erlang时，假设系统中的挂钟时间显示出与时间定义完全相同的速度向前移动的单调时间。这或多或少意味着原子钟（或更好的时间源）预计将附加到您的硬件上，并且预计硬件将被永远锁定在任何人类修补之外。虽然这可能是一个引人注目的想法，但事实并非如此。

“正常”的现代计算机不能保留时间，不能保留时间，除非你有一个芯片级原子钟连接到它。通常必须纠正您的计算机所感知的时间。因此，网络时间协议（NTP）协议与ntpd进程一起，尽力使您的计算机时间与正确的时间保持同步。在NTP校正之间，通常使用比原子钟更不强大的时间管理器。

但是，NTP不是故障安全的。 NTP服务器可能不可用，ntp.conf可能配置错误，或者您的计算机有时可能与Internet断开连接。此外，您可以让一个用户（甚至是系统管理员）认为处理夏令时的正确方法是每年两次调整时钟一次（这是不正确的方法）。为了使事情进一步复杂化，该用户从Internet获取了您的软件，并没有考虑计算机所感知的正确时间。用户不关心使挂钟与正确的时间保持同步。用户希望您的程序能够无限制地了解时间。

大多数程序员还希望时间可靠，至少在他们意识到工作站上的挂钟时间已经关闭一分钟之前。然后他们将它设置为正确的时间，但很可能不是顺利的方式。

当您总是希望系统上的挂钟时间正确时出现的问题数量可能是巨大的。因此，Erlang多年前就引入了“时间校正估计”或“时间校正”。时间校正依赖于大多数操作系统具有某种单调时钟的事实，实时扩展或某些独立于挂钟设置的内置“滴答计数器”。这个计数器可以具有微秒级的分辨率或更低的分辨率，但它有一个不容忽视的漂移。

## 时间校正
在时间校正开启的时候，erlang运行系统通过系统时间，系统单调递增时间来校正erlang单调递增时钟的频率，时间校正确保Erlang单调时间不会扭曲，并且频率相对准确。
频率调整的类型取决于所使用的时间扭曲模式。请到时间扭曲模式看更更多细节

默认情况下，如果在特定平台上存在时间校正，则启用时间校正。 对它的支持包括OS提供的OS单调时间和使用OS单调时间的Erlang运行时系统中的实现。 要检查您的系统是否支持OS单调时间，调用API ```erlang:system_info(os_monotonic_time_source)```.
要检查您的系统是否开启了时间校正，调用API```erlang:system_info(time_correction).```

通过启动参数```erl +c [true|false]```来开启或者关闭时间校正

如果禁用时间校正，则Erlang单调时间可以向前或向停止，甚至长时间冻结。 因此无法保证Erlang单调时钟的频率是准确或稳定的。

***您通常不想禁用时间校正*** 以前，性能损失与时间校正有关，但现在通常是另一种方式。 如果禁用时间校正，则可能会导致可扩展性差，性能不佳以及测量时间不佳。

## 时间扭曲安全代码(Time Warp Safe Code)
时间扭曲安全代码可以处理erlang系统时间，
```erlang:now()```在时间扭曲模式下表现不佳，在系统向后调整的时候，```erlang:now()```甚至可以卡主(freeze)知道系统时间到最后一次调用```erlang:now()```的地方，卡住可能会持续一段时间

```erlang:now()```在时间扭曲模式下，不一定是不安全的，如果你不用来获取时间，从性能和可伸缩性的角度来看是不理想的。 所以你真的想用其他功能替换它的使用。 有关如何替换，请参考章节，如何使用新的时间API

## 时间扭曲模式(Time Warp Modes)
通过启动参数
```
erl +C [no_time_warp|single_time_warp|multi_time_warp]
```
来开启时间扭曲模式
### 不使用时间扭曲模式(No Time Warp Mode)
时间偏移在运行时系统启动时确定，并且稍后不会更改。 这是默认行为，但不是因为它是最佳模式（它不是）。 它是默认的，因为这是运行时系统在ERTS 7.0之前的行为方式。 确保在启用其他模式之前，在时间扭曲期间可以执行的Erlang代码是时间扭曲安全的。

由于时间偏移不允许改变，时间校正必须调整Erlang单调时钟的频率，以使Erlang系统时间与OS系统时间平滑对齐。 这种方法的一个重要缺点是，如果需要调整，我们故意在Erlang单调时钟上使用错误频率。 此错误可能高达1％。 此错误将显示在运行时系统中的所有时间测量中。

如果未启用时间校正，则当OS系统时间向后跳跃时，Erlang单调时间会冻结。 单调时间的冻结持续到OS系统时间赶上。 冻结可以持续很长时间。 当OS系统时间向前飞跃时，Erlang单调时间也跃升了。

### 单时间扭曲模式(Single Time Warp Mode)
从引入开始，该模式或多或少是向后兼容模式。

在嵌入式系统上，当系统关闭时，系统没有电源，甚至没有电池也是不常见的。 这种系统上的系统时钟通常在系统启动时关闭。 如果没有使用时间扭曲模式，并且在更正OS系统时间之前启动Erlang运行时系统，则Erlang系统时间可能会长时间，几个世纪甚至更长时间内出错。

如果您需要使用非时间扭曲安全的Erlang代码，并且需要在更正OS系统时间之前启动Erlang运行时系统，则可能需要使用单时间扭曲模式。

*** 注意：***
使用此模式执行时间扭曲不安全代码的时间有限制。 如果只能使用时间扭曲安全代码，则最好使用多次扭曲模式(multi time warp mode)。

使用单时间扭曲模式，时间偏移分两个阶段处理：
#### 初步阶段
此阶段在运行时系统启动时启动。 确定基于当前OS系统时间的初步时间偏移。 从现在开始，这个偏移在整个初步阶段得到修复。

如果启用了时间校正，则调整Erlang单调时钟以使其频率尽可能正确。 但是，没有进行任何调整以尝试对齐Erlang系统时间和OS系统时间。 也就是说，在初步阶段，Erlang系统时间和OS系统时间可以彼此不同，并且没有尝试防止这种情况。

如果禁用时间校正，则OS系统时间的变化会影响单调时钟，与使用非时间扭曲模式时相同。

##### 最终阶段
当用户通过调用erlang来完成时间偏移时，此阶段开始：```erlang:system_flag(time_offset，finalize)```。 最终确定只能执行一次。
在最终确定期间，时间偏移被调整并固定，以便当前的Erlang系统时间与当前OS系统时间一致。 由于时间偏移可能在最终确定期间发生变化，因此Erlang系统时间可以在此时进行时间扭曲。 时间偏移从现在开始固定，直到运行时系统终止。 如果启用了时间校正，则从现在开始的时间校正也会进行调整，以使Erlang系统时间与OS系统时间保持一致。 当系统处于最后阶段时，它的行为与无时间扭曲模式完全相同。

为了使其正常工作，用户必须确保满足以下两个要求：

##### 前进时间扭曲(Forward Time Warp)
在确定时间偏移时所做的时间扭曲只能在没有遇到问题的情况下向前进行。 这意味着用户必须确保在启动Erlang运行时系统之前将OS系统时间设置为更早或等于实际POSIX时间的时间。

如果您不确定OS系统时间是否正确，请将其设置为在启动Erlang运行时系统之前保证早于实际POSIX时间的时间，这样才是安全的。

##### 完成正确的OS系统时间
当用户完成时间偏移时，OS系统时间必须正确。

如果未满足这些要求，系统可能表现得非常糟糕。

假设满足这些要求，则启用时间校正，并且使用诸如NTP的时间调整协议来调整OS系统时间，仅需要对Erlang单调时间进行小的调整以在完成之后保持系统时间对齐。 只要系统未挂起，所需的最大调整就是插入（或删除）闰秒。

***警告***
要使用此模式，请确保将在两个阶段中执行的所有Erlang代码都是时间扭曲安全的。

仅在最后阶段执行的代码不必能够应对时间扭曲。

### 多时间扭曲模式
***多时间扭曲模式与时间校正相结合是首选配置***
这就像Erlang运行时系统在几乎所有平台上都具有更好的性能，更好的扩展性和更好的表现。此外，时间测量的准确性和精确度更好。只有在古代平台上执行的Erlang运行时系统才能从另一种配

时间偏移可以随时改变而没有限制。也就是说，Erlang系统时间可以随时执行向前和向后的时间扭曲。当我们通过改变时间偏移将Erlang系统时间与OS系统时间对齐时，我们可以启用时间校正，试图将Erlang单调时钟的频率调整为尽可能正确。这使得使用Erlang单调时间的时间测量更加准确和精确。

如果禁用时间校正，则如果OS系统时间跳跃，则Erlang单调时间会向前跳跃。如果OS系统时间向后跳跃，Erlang单调时间会短暂停止，但它不会长时间冻结。这会更改时间偏移，以使Erlang系统时间与OS系统时间保持一致。

*** 警告 ***
要使用此模式，请确保将在运行时系统上执行的所有Erlang代码都是时间扭曲安全的。

## 新时间API
旧时API基于```erlang：now/0```。 ```erlang：now / 0```旨在用于许多无关的事情。 这将这些不相关的操作捆绑在一起，并导致不需要出现此类问题的操作的性能，可伸缩性，准确性和精度问题。 为了改善这一点，新API在多个功能上传播不同的功能。

为了向后兼容，```erlang：now / 0```保持“原样”，但强烈建议您不要使用它。 erlang的许多用例：```now / 0```阻止你使用新的多次扭曲模式，这是新时间功能改进的重要部分。

某些系统上的一些新BIF（可能令人惊讶地）在新启动的运行时系统上返回负整数值。 这不是错误，而是内存使用优化。

新的时间API

```
erlang:convert_time_unit/3

erlang:monotonic_time/0

erlang:monotonic_time/1

erlang:system_time/0

erlang:system_time/1

erlang:time_offset/0

erlang:time_offset/1

erlang:timestamp/0

erlang:unique_integer/0

erlang:unique_integer/1

os:system_time/0

os:system_time/1
```
新API还包含以下现有BIF的扩展：
```
erlang:monitor(time_offset, clock_service)

erlang:system_flag(time_offset, finalize)

erlang:system_info(os_monotonic_time_source)

erlang:system_info(os_system_time_source)

erlang:system_info(time_offset)

erlang:system_info(time_warp_mode)

erlang:system_info(time_correction)

erlang:system_info(start_time)

erlang:system_info(end_time)
```

### 新的Erlang单调时间
Erlang单调时间是从ERTS 7.0开始的新时代。引入它来分离时间测量值，例如从日历时间开始经过的时间。在许多使用情况中，需要测量经过的时间或指定相对于另一时间点的时间，而不需要知道UTC中涉及的时间或任何其他全局定义的时间尺度。通过引入具有本地定义的时间尺度，可以在该时间尺度上管理与日历时间无关的时间。 Erlang单调时间使用这样的时间尺度和局部定义的开始。

Erlang单调时间的引入允许我们分别调整两个Erlang时间（Erlang单调时间和Erlang系统时间）。通过这样做，经过时间的准确性不必仅仅因为系统时间在某个时间点发生错误而受到影响。两次的单独调整仅在时间扭曲模式中执行，并且仅在多次扭曲模式中完全分离。除了多时间扭曲模式之外的所有其他模式都是出于向后兼容的原因。当使用这些模式时，Erlang单调时间的准确性受到影响，因为在这些模式下Erlang单调时间的调整或多或少地与Erlang系统时间相关联。

系统时间的调整可能比使用时间扭曲方法更加平稳，但我们认为这将是一个糟糕的选择。由于我们可以通过使用Erlang单调时间来表达和测量与日历时间无关的时间，因此最好立即公开Erlang系统时间的变化。这是因为在系统上执行的Erlang应用程序可以尽快对系统时间的变化作出反应。这也或多或少是大多数操作系统处理此问题的方式（OS单调时间和OS系统时间）。通过平滑地调整系统时间，我们只会隐藏系统时间发生变化的事实，并使Erlang应用程序更难以以合理的方式对变化作出反应。

为了能够对Erlang系统时间的变化作出反应，您必须能够检测到它发生了。当前时间偏移更改时，会发生Erlang系统时间的更改。因此，我们介绍了使用erlang监视时间偏移的可能性：```monitor（time_offset，clock_service）```。当时间偏移更改时，监视时间偏移的进程将使用以下格式发送消息：

```
{'CHANGE', MonitorReference, time_offset, clock_service, NewTimeOffset}
```

### 唯一值（Unique Values）
除了报告时间，```erlang：now / 0```还会产生唯一且严格单调递增的值。 为了从时间测量中分离这个功能，我们引入了```erlang：unique_integer（）```。


### 如何使用新时间API

##### 获得系统时间(Retrieve Erlang System Time)
不要使用```erlang:now()```,使用```erlang:system_time()```

##### 测量经过的时间(Measure Elapsed Time)
不要使用两次```erlang:now()```然后使用```timer:now_diff/2```来计算

使用 ```erlang:monotonic_time/0```来获取时间戳，然后两次来计算差值，结果是```native time unit```,然后使用```erlang:convert_time_unit/3```来计算差值

更方便的方式是使用```erlang:monotonic_time/1```来计算，但是会丢失一部分精度和准确度

##### 确定事件顺序（Determine Order of Events）
不要通过保存```erlang:now/0```一系列值来达到目的

通过保存 ```erlang:unique_integer([monotonic])``` 一些列值来判断顺序

##### 确定事件顺序与事件的时间（Determine Order of Events with Time of the Event）
通过在事件发生时使用```erlang：now / 0```保存时间戳来确定事件的顺序。

```
Time = erlang:monotonic_time(),
UMI = erlang:unique_integer([monotonic]),
EventTag = {Time, UMI}
> {-576460097339893645,-576460752303423481}
```

根据创建时间，这些元组在当前运行时系统实例上严格单调排序。 重要的是单调时间在第一个元素中（比较两元组时最重要的元素）。 使用元组中的单调时间，您可以计算事件之间的时间。

如果您对事件发生时的Erlang系统时间感兴趣，还可以使用```erlang：time_offset / 0```保存事件之前或之后的时间偏移。 添加时间偏移的Erlang单调时间对应于Erlang系统时间。

如果您在时间偏移可以改变的模式下执行，并且您希望在事件发生时获得实际的Erlang系统时间，则可以将时间偏移量保存为元组中的第三个元素（比较三个元素时最不重要的元素 - 元组）。

##### 创建一个唯一的名称
使用从```erlang：unique_integer / 0```返回的值在当前运行时系统实例上创建唯一的名称。 如果你只想要正整数，你可以使用```erlang：unique_integer（[positive]）```。

##### 具有唯一值的种子随机数生成
使用```erlang：monotonic_time（），erlang：time_offset（），erlang：unique_integer（）```和其他功能的组合生成种子随机数。

## 支持新旧OTP版本

可能要求您的代码必须在不同OTP版本的各种OTP安装上运行。 如果是这样，您就无法使用开箱即用的新API，因为它在OTP 18之前的版本中不可用。解决方案是不要避免使用新API，因为您的代码将无法从可伸缩性和准确性改进中受益 制作。 相反，在可用时使用新API，并在新API不可用时返回```erlang：now / 0```。

幸运的是，除了以下内容之外，大多数新API都可以使用现有基元轻松实现：

```
erlang:system_info(start_time)
erlang:system_info(end_time)
erlang:system_info(os_monotonic_time_source)
erlang:system_info(os_system_time_source))
```
通过使用回退到erlang上的函数包装API：now / 0当新API不可用时，并使用这些包装而不是直接使用API，问题就解决了。 例如，这些包装器可以在```$ ERL_TOP / erts / example / time_compat.erl```中实现。
